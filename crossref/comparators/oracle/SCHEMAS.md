## Extraction Method

The `*-body.xsd` files are generated by `extract-schemas.py` from the WSDLs in `onvif-server/wsdl/`.

**Script**: `crossref/comparators/oracle/extract-schemas.py`
**Run**: `python3 extract-schemas.py <wsdl_dir> <output_dir>` (python3 stdlib only, no deps)

For each service WSDL, the script:
1. Finds the `<xs:schema>` element inside `<wsdl:types>`
2. Extracts the raw XML block (depth-tracked, handles nested elements)
3. Copies ALL in-scope namespace declarations from `wsdl:definitions`, `wsdl:types`, and `xs:schema`
   onto the extracted `<xs:schema>` element â€” critical for prefix resolution (e.g. `tt:`, `tds:`, `tptz:`)
4. Rewrites `schemaLocation` attributes to bare local filenames (offline resolution)
5. Writes as `<service>-body.xsd`

The script also copies the shared XSDs with schemaLocation rewrites and applies a
determinism fix to `onvif.xsd`: `namespace="##targetNamespace"` â†’ `namespace="##other"`
on `xs:any` wildcards. This resolves the non-deterministic content model error that
strict validators (xmllint/libxml2) report; Xerces (Java) is also compatible with this.

The Dockerfile runs the script during the build stage (after `mvn dependency:go-offline`,
before `mvn package`) to ensure the jar always bundles freshly-extracted schemas.

| Namespace | Prefix | Status |
|-----------|--------|--------|
| `http://docs.oasis-open.org/wsn/bw-2` | `wsntw` | WSDL-level import only; NOT in `xs:schema`; no `xs:import` needed |
| `http://docs.oasis-open.org/wsrf/rw-2` | `wsrf-rw` | WSDL-level import only; NOT in `xs:schema`; no `xs:import` needed |
| `http://www.w3.org/2006/05/addressing/wsdl` | `wsaw` | Policy annotation prefix; NOT in `xs:schema`; no `xs:import` needed |

These namespaces appear in the namespace declarations injected onto `events-body.xsd`
(because they are in-scope on `wsdl:definitions`) but no schema content from them is
referenced in the extracted `<xs:schema>` element. The resolver ignores unresolvable
namespaces (returns null), which is correct â€” Xerces silently skips imports with no
matching content when `processContents="lax"` is in effect.

## Offline Resolution Mechanism

Two complementary mechanisms ensure zero network access:

1. **schemaLocation rewriting**: `extract-schemas.py` rewrites every remote `schemaLocation`
   to a local bare filename (e.g. `https://www.w3.org/2005/05/xmlmime` â†’ `xmlmime.xsd`).
   This satisfies validators that follow schemaLocation hints literally.

2. **LSResourceResolver**: `OnvifSchemaResolver` in Oracle.java maps each import namespace URI
   and common filename to a classpath resource under `/schemas/`. This handles any validator
   that ignores schemaLocation and resolves by namespace URI alone, as well as indirect
   resolutions (e.g., `common.xsd` is `xs:include`d by `onvif.xsd` â€” the resolver handles
   the bare filename `common.xsd` via its filename map).
